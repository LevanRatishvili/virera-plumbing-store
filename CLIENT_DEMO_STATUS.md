# კლიენტის დემო სტატუსი

## მზად არის დემოსთვის

- საჯარო კლინიკის გვერდი: მთავარი ბლოკი, სლაიდერი, სერვისები, ექიმები, ფასები, საკონტაქტო ინფორმაცია, რუკა, სოციალური ბმულები და ფუტერი.
- ვიზიტის მოთხოვნის მოკლე ფორმა მინიმალური ველებით: სახელი, ტელეფონი, სერვისი, ექიმი, სასურველი თარიღი/დრო და არასავალდებულო კომენტარი.
- დაცული ადმინ პანელი პაროლით.
- ჩაწერის მოთხოვნების სია, ფილტრები და სტატუსის შეცვლა ადმინში.
- საიტის კონტენტის რედაქტორი ძირითადი ტექსტებისთვის, სლაიდერისთვის, სერვისებისთვის, ექიმებისთვის, ფასებისთვის, კონტაქტისთვის და სამართლებრივი/თანხმობის ტექსტებისთვის.
- მედია არჩევის პანელი არსებული კლინიკის სურათებით.
- კონტენტის JSON ექსპორტი და იმპორტი.
- კლინიკის ასისტენტი ზოგად კითხვებზე; სამედიცინო რჩევას არ იძლევა.

## ადმინს შეუძლია

- ნახოს ახალი ვიზიტის მოთხოვნები.
- შეცვალოს მოთხოვნის სტატუსი.
- განაახლოს საჯარო ტექსტები და ჩამონათვალი.
- შეინახოს, დააბრუნოს საწყის მნიშვნელობებზე, ექსპორტი გააკეთოს და იმპორტით აღადგინოს კონტენტი.
- ნახოს storage სტატუსი და გაიგოს, ჩართულია თუ არა მუდმივი საცავი.

## დემო მონაცემებია

- კლინიკის სახელი, მისამართი, ტელეფონი და ელფოსტა.
- ექიმების სახელები, პროფილები და ფოტოები.
- სერვისების აღწერები და ფასები.
- რუკის მისამართი და სოციალური ბმულები.
- ფუტერის, თანხმობისა და პერსონალური მონაცემების ტექსტები.

## კლიენტისგან საჭიროა

- საბოლოო კლინიკის სახელი და საკონტაქტო მონაცემები.
- რეალური ექიმების სია, ფოტოების ნებართვები და სპეციალობები.
- რეალური სერვისები, ფასი ან ფასის პოლიტიკა.
- საბოლოო მისამართი და Google Maps ბმული/ჩასმა.
- სამართლებრივი ტექსტების დადასტურება: პერსონალური მონაცემები, თანხმობა და ფუტერი.
- სოციალური ქსელების ოფიციალური ბმულები.

## საცავის შეზღუდვა

ამ ეტაპზე production გარემოში მუდმივი storage არ არის კონფიგურირებული. SQLite ფაილი შეიძლება შეიცვალოს deploy-ის ან გარემოს გადატვირთვისას. დიდი ცვლილებების წინ ადმინმა უნდა ჩამოტვირთოს JSON backup. საბოლოო ჩაბარებამდე საჭიროა persistent disk, გარე მონაცემთა ბაზა ან სხვა შეთანხმებული მუდმივი საცავი.

## უსაფრთხოების საზღვრები

- ფორმა არ ითხოვს დიაგნოზს, პირად ნომერს, სამედიცინო ისტორიას, რეცეპტებს, ფაილებს ან სხვა მგრძნობიარე სამედიცინო მონაცემებს.
- ასისტენტი არ სვამს დიაგნოზს და არ იძლევა მკურნალობის, მედიკამენტის ან გადაუდებელი შეფასების რჩევას.
- ადმინის API დაცულია ავტორიზაციით.
- backup ფაილი შეიცავს მხოლოდ საიტის კონტენტს და build ინფორმაციას; პაროლები, სესიები და ჩაწერის მოთხოვნები არ შედის.
- ძველი ecommerce/product/order/chat API ზედაპირები დემოსთვის გამორთულია.

## საბოლოო production ჩაბარებამდე

- Render-ზე კონფიგურირებული უნდა იყოს ძლიერი `ADMIN_PASSWORD`.
- უნდა გადაწყდეს მუდმივი storage-ის ვარიანტი.
- კლიენტმა უნდა დაამტკიცოს ყველა საჯარო ტექსტი და ფოტო.
- საჭიროების შემთხვევაში უნდა დაემატოს რეალური privacy/legal ტექსტები იურისტის ან პასუხისმგებელი პირის მიერ.
- უნდა განისაზღვროს, ვინ მართავს admin backup-ს და რა სიხშირით.

## რეკომენდებული შემდეგი ნაბიჯები

4. საბოლოო უსაფრთხოების შემოწმება რეალური პაროლით და production storage-ით.
5. ფინალური handoff ინსტრუქციის მომზადება ადმინისთვის.
